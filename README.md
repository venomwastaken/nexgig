# NexGiG

Student freelancing marketplace. Backend: FastAPI + SQLModel + PostgreSQL.
Frontend: React + Vite + TypeScript.

## First-time setup (after cloning)

Prerequisites: Docker Desktop and Node.js installed on your machine.

1. `git clone <repo-url>` then `cd nexgig`
2. `cp backend/.env.example backend/.env`
3. `docker compose up` — starts Postgres + FastAPI together
   - Confirm it worked: visit http://localhost:8000/health, should show `{"status": "ok"}`
   - Leave this running in its own terminal tab
4. In a **new terminal tab**: `cd frontend`
5. `cp .env.example .env`
6. `npm install --save-exact` — keeps versions locked to what's in package.json (see "Frontend gotchas" below for why)
7. `npm run dev` — starts the frontend, usually at http://localhost:5173
   - Confirm it worked: open that URL, should show "Backend status: ok"
   - If it says "backend unreachable," make sure step 3 is still running

You do **not** need PostgreSQL installed directly — Docker handles that. You **do** need a local Python venv if you're touching the backend (see below), because Alembic and any backend scripts run from your host machine, not inside the container.

### Backend Python environment (one-time)

From `backend/`:

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

If PowerShell blocks the activation script, run this once (as admin or per-user):
```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

## Day-to-day cheatsheet

### Everyone — daily startup
```
docker compose up          # backend + db (own terminal tab, leave running)
cd frontend && npm run dev # frontend (separate terminal tab)
```
Visit http://localhost:5173 — should say "Backend status: ok".

### Frontend group

```powershell
cd frontend
npm run dev                 # start dev server
npm install <pkg> --save-exact   # ALWAYS use --save-exact when adding packages
```

**Why `--save-exact`:** Clerk's React peer dependency range starts at `~19.0.3`. If React is pinned exactly at `19.0.0` (no caret), you'll get peer dependency conflicts. `.npmrc` already has `save-exact=true` set repo-wide, but pass the flag explicitly if you're ever unsure.

**Don't** upgrade Vite past 5.x without checking with the team first — Vite 8's Rolldown/Oxc internals break `@vitejs/plugin-react`. We're intentionally pinned to `5.4.x`.

### Backend group

```powershell
cd backend
.\venv\Scripts\Activate.ps1     # activate venv FIRST, every session
```

Once activated, your prompt should show `(venv)`. All backend commands (alembic, running scripts, installing packages) should happen with the venv active.

```powershell
pip install <pkg>
pip freeze > requirements.txt   # if you add a new dependency
```

### Database group — Alembic migrations

**Always activate the venv first.** Alembic runs against `localhost` (not the Docker service name `db`) because it's running from your host machine, not inside the container.

```powershell
cd backend
.\venv\Scripts\Activate.ps1

# after changing a SQLModel model:
alembic revision --autogenerate -m "short description of the change"

# review the generated file in backend/alembic/versions/ before applying!
alembic upgrade head
```

**Checklist before running `alembic upgrade head`:**
- [ ] venv is activated (see `(venv)` in prompt)
- [ ] `docker compose up` is running (Postgres needs to be up for Alembic to connect)
- [ ] `DATABASE_URL` in `backend/.env` points to `localhost` (not `db`) — that's the host-side value; the containerized backend gets its own `DATABASE_URL` from `docker-compose.yml`'s `environment:` block, which overrides `.env`
- [ ] Reviewed the autogenerated migration file — SQLModel autogenerate sometimes needs manual fixes

**One-time fix already applied:** `alembic/script.py.mako` has `import sqlmodel` added, since autogenerated migrations reference `sqlmodel` without importing it by default. If you regenerate `script.py.mako` for any reason, re-add that import.

**Inspecting the DB directly:**
```powershell
docker compose exec db psql -U <user> -d <dbname>
```
(check `backend/.env` for the actual user/db name)

Common psql commands once inside:
```sql
\dt            -- list tables
\d "usertable" -- describe a table
\q             -- quit
```

### Auth (Clerk) — if you're touching auth code

- Clerk is the identity source of truth. Don't add fields to `UserAccount` that Clerk already owns (password, email verification status, last login).
- JWTs are verified via `PyJWKClient` — never trust an `X-Clerk-User-Id`-style header directly.
- If a Clerk custom claim isn't showing up in the JWT payload, double check you edited the claim in the correct place in the Clerk dashboard (this has bitten us before).

## Project structure
- `backend/` — FastAPI service. See `backend/app/main.py` for the entrypoint.
- `frontend/` — React app. See `frontend/src/main.tsx` for the entrypoint.
- `docker-compose.yml` — runs backend + Postgres together for local dev.
- `.github/workflows/ci.yml` — runs automatically on every PR.

## Workflow
- Branch off `main` for every change: `feature/short-description`
- Open a PR into `main` once done — CI must pass before merging
- No direct pushes to `main`
- **CI note:** GitHub Actions runs the workflow file from your PR branch on `pull_request` events, not from `main`. If you push a fix to the workflow itself, you need a new commit on the branch to trigger a fresh run — re-running an old job just replays the old snapshot.

## Troubleshooting quick reference

| Symptom | Likely cause |
|---|---|
| Frontend says "backend unreachable" | `docker compose up` isn't running |
| Alembic can't connect to DB | Postgres container isn't up, or `DATABASE_URL` still points to `db` instead of `localhost` in your local `.env` |
| `email NOT NULL` constraint errors | Migration wasn't run — activate venv, `alembic upgrade head` |
| 401 on API calls during testing | Malformed/missing `Authorization: Bearer <token>` header |
| npm install fails with peer dep errors | Missing `--save-exact`, or React/Clerk version drift — check `.npmrc` |
| `ResponseValidationError` from FastAPI | Response model likely references a field that no longer exists (e.g. a stale field removed from `UserAccount`) |
